# Implementation Plans

Generated by the `improve` skill (direction audit, `next`) on 2026-06-15,
against commit `9f0b076`. Trajectory chosen by the maintainer: **make FUN26 a
reusable, config-driven conference template.** Each executor: read the plan
fully before starting, honor its STOP conditions, and update your row when done.

## Context the audit established

The June 2026 merge already shipped most of what an earlier audit had proposed:

- **Session resources** (slides/video/code/link) — schema, detail-page
  rendering, and an author guide all exist (`src/content.config.ts:24-33`,
  `docs/content.md:55-74`). **Nothing to build** — only content (real links)
  remains, added via markdown PRs. This is why there is no "resources" plan.
- **Speaker directory + profile pages** — `/speakers` index and
  `/speakers/[slug]` pages exist for day-2 speakers (`src/pages/speakers/`).
  Day-1 speakers intentionally use generated SVG avatars and have no profiles
  (provenance-blocked, see `docs/content.md:27-53`). The remaining codeable gap
  is a **content-integrity guard** (plan 004), not the directory itself.
- **Multi-day schedule** — the day switcher and per-day stages already derive
  from session dates (`src/components/schedule/ScheduleApp.tsx:51-60`). Plan 001
  finishes the template job by extracting the remaining hardcoded event identity.
- **Language taxonomy** — the `language` field exists and is _displayed_, but
  never _filtered on_. Plan 002 adds the missing filter/search UI.

Shared-state features (live Q&A, ratings, favorite counts, cross-device sync)
are deliberately deferred behind a backend decision —
`docs/decisions/0001-backend-surrealdb-linkedin-auth.md` (status: proposed).
None of the plans below require a backend; all stay within the static +
localStorage architecture.

There is **no test runner** in this repo (no `test` script in `package.json`).
Verification gates are `pnpm typecheck`, `pnpm lint`, `pnpm build`, targeted
`grep`, and a manual `pnpm preview` smoke check — the plans use these, not a
test framework.

## Dependency notes

- No plan _hard_-depends on another. Recommended order is 001 → 002 → 003 → 004.
- 001 is **soft-before** 002 and 003: it introduces `src/event.config.ts` and
  changes page-title strings and `src/layouts/main.astro`. 002 and 003 both add
  new pages/UI that should read the new config and (003) add a header nav link
  to the same `main.astro` header. Doing 001 first means 002/003 build on the
  config instead of re-hardcoding strings, and avoids editing `main.astro`
  twice. If 002/003 run before 001, they may hardcode the event name (acceptable;
  001's drift check will then flag those strings).
- 004 is fully independent and can run at any time.

## Findings considered and rejected

- **D4 — Post-event resources (recordings/slides):** already implemented end to
  end (schema + detail page + author docs). No code work remains; adding actual
  links is content maintenance, not a plan.
- **D3 — Build the speaker directory/pages:** already implemented for day-2.
  Re-scoped to the integrity guard (plan 004). Day-1 speaker profiles are
  content work blocked on official-site access (`docs/content.md:48-53`), not a
  code plan.
- **Cross-device favorites/notes sync, live Q&A, ratings, favorite counts:**
  deferred by `docs/decisions/0001-…` pending the backend decision. Out of
  scope for the static template.
- **Parameterizing `fun26.*` localStorage keys / `.ics` `PRODID`/`UID` /
  notification tag prefixes:** intentionally left as a fixed `fun26.` namespace
  in plan 001 — renaming storage keys would silently drop existing users'
  favorites/notes/theme, and these identifiers are not user-visible. Only
  user-facing text and theming get centralized.
